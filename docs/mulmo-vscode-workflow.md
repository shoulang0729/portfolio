# Mulmo ▶ VS Code 開発フロー（このリポジトリの作り方）

このドキュメントは **VS Code 上の AI コーディングエージェント（Claude Code 等）が最初に読む** ためのものです。
このリポジトリ（portfolio アプリ）が「誰の指示で・どういう流れで」開発されているかを説明します。

---

## 登場人物

- **Mulmo（MulmoClaude）** = オーナー Toshio の手元で動く上流アシスタント。
  要件整理・調査・設計・実装方針の決定を担当する「設計者」。コードは原則書かない。
  知識ベース（Wiki／メモリ）と投資システムの全体像を持っている。
- **VS Code のエージェント（あなた）** = 下流の「実装者」。
  Mulmo が作った設計書・指示書を受け取り、実際のコーディング・git 操作・テスト・デバッグを担当する。

## 基本原則：設計 ▶ 実装は一方通行

```
Toshio → Mulmo（設計・指示書）→ VS Code（実装）→ PR → Toshio レビュー
```

- 流れは **設計 ▶ 実装の一方向**。あなた（実装者）が設計そのものを勝手に変えない。
- 実装中に「設計を変えた方がいい」と気づいたら、**自己判断で変更せず一旦手を止めて報告**する。
  設計変更は Mulmo に戻して指示書を更新し、新版を受け取ってから再開する（これがドリフト防止の要）。
- あなたの成果物は **差分（PR）**。取り込みは下記「マージ方針」に従う。

### マージ方針（取り込み・2026-06-21 確定）

**実装 PR の取り込みは VS Code が owner。** 設計レーン（Mulmo）は実装レーンに"戻り込まない"＝完全一方向。どちらのモードも CI（品質ゲート）green が大前提。自動マージは"実装の取り込み"であって"設計変更"の承認ではない。

- **① 自動マージ（既定・低〜中リスク）** — docs・データ投入・handoff doc でスコープ確定済みの実装は、CI green なら **VS Code が自身で squash マージ＋ブランチ削除**（CLAUDE.md の autonomous merge）。**Toshio は事後レビュー**し、イマイチなら Issue を起票して差し戻す（review-after）。
- **② 事前レビュー（大規模・破壊的・load-bearing）** — 影響が広い／ロールバックしづらい／load-bearing を触る変更は、**Toshio の事前目視レビュー後にマージ**。迷ったらこちら。
- **Mulmo はコード PR をマージしない。** Mulmo の役割は「設計供給」＋「**完了把握（read-only モニタリング）**」＝定期的に Issue / PR / commit を読み、実装完了を検知したら設計正本（handoff doc）に完了スタンプ＋Wiki 更新＋次の設計を供給。判断が要るもの・CI red の滞留だけ Toshio へエスカレーション。
- ※Mulmo 自身の `docs/` PR（設計レーン内）は Mulmo が管理してよい＝実装レーンへの"戻り"ではない。
- **Mulmo はコード（`src/` / `scripts/` / テスト等）を直接書かない・修正しない（自明な1行でも）。** コード修正が要るときは Issue 化し、**ユーザーに「Issue #XXX を VS Code に投げて」と明示リマインドする**。VS Code は**ユーザーが起動して Issue を渡した時だけ**動く＝**dispatch のトリガーはユーザー**。Mulmo が代わりに書くと一方通行が濁る（2026-06-23 確定。例外的 hotfix も今後しない）。

## 最初に読むもの（この順番で）

1. **その回の指示書** — `docs/handoff/<YYYY-MM-DD>-<feature-slug>.md`
   そのタスク専用のエントリポイント。背景・対象ファイル・手順・受け入れ条件・触ってはいけない範囲が書いてある。
2. **リポの常設ルール（毎回必読）**
   - `CLAUDE.md` — プロジェクト規約・エージェント向け運用ルール
   - `.claude/codex-review-prompt.md` — レビュー観点
3. **触る範囲の正本**（指示書が都度指定する）
   - 例: Briefing → `docs/briefing-generation-spec.md` / MF 取り込み → `mf-import-config.json` の `schema`

## 守ること（要点・詳細は CLAUDE.md）

- **1 Task = 1 ブランチ = 1 PR = 1 Issue**。ベースは必ず `main`、push 前に `git pull --rebase`。
- **Issue の発行は設計レーンの行為**。実装中に設計側の問題（生成プロンプトのドリフト・仕様の矛盾等）に気づいたら、実装者は新 Issue を起こさず**元 Issue のコメント欄に報告**する。別 Issue への切り出しは設計側（Mulmo / Toshio）が判断する（2026-07-06 追加・#540 の経緯から）。
- **squash マージのみ＋ブランチ auto-delete**。feature ブランチは残さない。
- バージョン: `index.html` の `?v=YYYYMMDDX` を CSS / JS / SW 全部揃えて bump。`dist/app.js` は CI 自動ビルド（手 commit 不可）。
- 品質ゲートを green に: `vitest` / `eslint` / `prettier` / `check:types` / `check:circular` / `e2e`。
- data-action 委譲・`escapeHTML` 必須・色は CSS 変数のみ・`!important` 禁止。
- **`assets/*.css` に `prettier --write` を掛けない**（全体再整形で巨大 diff 化する）。
- load-bearing なフィールド名は変更しない（例: mf-holdings の `cat` / `cur` / `value` / `totals.imported` / `asOf`）。
- 勝手にやらない（要確認）: force push / `reset --hard` / main 削除 / Secrets / `CLAUDE.md` / `.claude` 設定 / ワークフロー大改修。

## Mulmo ↔ VS Code のやり取りチャネル原則

**「生き続けるもの → docs / 消費されるもの → Issue」**

| チャネル | 寿命 | 使う場面 |
|---|---|---|
| `docs/` のMarkdownファイル | 永続 | アーキテクチャ・設計制約・API仕様・調査結果の長期参照 |
| GitHub Issue | 完了で閉じる | 調査依頼・実装タスク・バグ報告・「やること」の依頼と完了確認 |

**運用パターン:**

1. Mulmo が `docs/` に設計コンテキスト（背景・方針・制約）を書く
2. Mulmo が Issue を起票し、関連 doc へリンク（「背景は doc 参照」）
3. VS Code が Issue コメントに調査結果・完了報告を書いて Issue をクローズ
4. 結果を長期参照したい場合のみ doc に転記

**やりがちなアンチパターン:**

- 仕様書に調査依頼と結果記入欄を同居させる → 完了が見えない・PR との連携ができない
- すべてを Issue のコメントで済ませる → 設計の背景が流れて消える

**補強（2026-06-21 合意）:**

1. **設計の差し戻し・再決定は「議論 → Issue / 確定判断 → doc 転記」**。
   実装中に設計矛盾が見つかった場合（例: spec の前提が実データと食い違う）、その *やり取り* は Issue/PR コメントで進めてよいが、*確定した判断* は必ず正本 doc に書き戻す。Issue コメントだけで決着させない（次に読む人に伝わらない）。
2. **Mulmo↔オーナーのチャットは VS Code から見えない**。
   設計判断の多くは Mulmo とオーナーのチャット（フォーム等）で確定する。VS Code はそれを参照できないので、**チャットで決めた設計は必ず doc か Issue に落とす**。「チャットで言ったはず」は無効。
3. **Issue は doc を指すだけ・設計内容を複製しない**。
   Issue 本文に設計の中身をコピペするとドリフトの種になる。Issue には「やること＋ doc へのリンク」だけを書き、設計の正本は常に doc 側に一本化する。

## 並行作業プロトコル（worktree-first・起動時の型／2026-06-21）

**目的**: 設計と実装を *同時に* 走らせても衝突しないようにする。
**禁止する型（A アンチパターン）**: 1 つのセッションが **メインの作業ツリーで最初から設計も実装も全部抱え込む**。これをやると (a) 他へディスパッチできない、(b) 設計と実装が同じツリーで直列化する、(c) ブランチ／作業ツリーが衝突する（残骸ブランチ事故の根）。

### 起動時の型（全セッション共通・コードに触る前に宣言する）

1. **レーンを宣言する** — `設計` / `実装` / `壁打ち（コード無し）` のどれか。
2. **専用 git worktree で開始する**（壁打ちレーンを除く）。
   **メインの共有作業ツリー（オーナー手元の clone）で 2 つ以上のセッションを同時に走らせない。**
   ```bash
   git worktree add ../portfolio-<feature-slug> -b <branch> origin/main
   ```
   - 設計レーン = `docs/` だけを触る worktree（コードは書かない）
   - 実装レーン = コードを触る worktree（handoff doc を 1 本消費）
3. 終わったら PR → squash マージ（「マージ方針」に従う）→ `git worktree remove`。残骸を残さない。

> **適用範囲**: worktree 隔離が必須なのは *1 つの clone を複数セッションが共有する* 場合の衝突防止。
> Mulmo（サンドボックス clone）と VS Code（オーナー手元 clone）は**別 clone ＝自然に隔離**されているので、各自は feature ブランチ運用で足りる（worktree は任意）。
> **同一 clone 内で 2 つ以上を同時に走らせる時のみ worktree 必須**。共有正本（`mf-holdings.json` / `positions.json` 等）は main 上の単一ソース＝feature ブランチで main を汚さない。

### レーン分担

| レーン | 担当 | 触る範囲 | 成果物 |
|---|---|---|---|
| 設計 | Mulmo（＋背景サブエージェント） | `docs/` のみ | handoff doc ＋ Issue |
| 実装 | VS Code | コード | PR |
| 壁打ち | Mulmo | なし（worktree 不要） | チャット → 確定は doc / Issue に落とす |

### 並行の効かせ方（設計も実装も止めない）

- **設計の並行化** — Mulmo は独立した設計タスクを **背景サブエージェント（worktree 隔離）で同時に複数** 出す。1 本ずつ直列に書かない。
- **実装の並行化** — VS Code は handoff doc 単位で **別ブランチ（別 worktree）を同時に** 進める。
- **設計 ▶ 実装は非同期** — 実装は設計の全完了を待たず、**出来上がった handoff から順に着手**。設計側は **数本まとめて前倒し供給**してバックログを枯らさない（＝ Mulmo がクリティカルパスから外れる）。
- 共有正本（`mf-holdings.json` / `positions.json` 等）は **main 上の単一ソース**。worktree は feature ブランチで main を汚さない。

### 起動時チェックリスト（このリポで作業を始める AI が最初に答える）

- [ ] 自分のレーンは？（設計／実装／壁打ち）
- [ ] いま他のセッションが走っている？ → 走っているなら **必ず専用 worktree** で開始したか？
- [ ] 設計レーンなら `docs/` 以外を触っていないか？
- [ ] 実装レーンなら対応する handoff doc は存在するか？（無ければ Mulmo に設計を依頼して止まる）

## 困ったとき

- 指示書と既存コードが矛盾する、正本が見つからない、設計判断が要る → **止めて Toshio に確認**。
  推測で実装を進めて PR を膨らませない。
