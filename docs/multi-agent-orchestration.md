# マルチエージェント実装ワークフロー（Haiku 委任ガイド）

複数モデル（Haiku / Sonnet / Opus）を使い分けて実装する際の運用ルール。
**トークン・時間を節約しつつ手戻りを防ぐ**ことが目的。

> このドキュメントはプロジェクト非依存の運用ノウハウです。他リポジトリにも
> そのままコピーして引き継げます。CLAUDE.md の「並列セッション/サブエージェント
> 運用ルール」と対になっています。

---

## 大原則：「決められたものを書く」が Haiku、「何を選ぶか決める」が Opus/Sonnet

| モデル | 向き | 例 |
|---|---|---|
| **Haiku** | 仕様が確定した機械的な記述 | テスト追加、JSDoc、curated データ追加、確定I/Fへの実装、CSS 微修正、ボイラープレート配線 |
| **Sonnet** | 中規模リファクタ・複数モジュール連携・環境構築 | IndexedDB 移行、E2E 導入、パーサ実装 |
| **Opus** | 設計判断・根本バグ修正・型/I/F 設計・衝突解消 | API契約設計、CORS/外部連携方針、レース修正、コンフリクト解消 |

判断を要するタスクを Haiku に丸投げすると、安易な選択をして手戻りが発生し、
**かえって高くつく**（過去事故: Haiku が typescript 最新版を入れて madge と衝突）。

---

## 推奨フロー：設計は上位モデル、量産は Haiku

```
1. Opus/Sonnet が「仕様」を固める
   - I/F 契約（関数シグネチャ・入出力の型・エラー方針）
   - 外部依存の方針（CORS 回避・データ形式・フォールバック順）
   - データ構造・キャッシュ鍵・命名
   ↓
2. 確定仕様を「実装タスク」に分解し Haiku サブエージェントへ委任
   - 1 タスク = 1 ファイル/1 機能 = 受け入れ条件を明記
   - 「この I/F に対して実装せよ。設計判断が必要になったら止めて報告」と指示
   ↓
3. Opus/Sonnet が結果をレビュー・統合・PR/マージ
```

**節約ポイント**: トークンを食う「実装の量産」を Haiku に流し、事故りやすい
設計判断は上位モデルが握る。Haiku に投げてよいのは *仕様が一意に決まっている*
チャンクだけ。

---

## サブエージェント運用の必須ルール

1. **ベースブランチは必ず `main`**。他エージェントのブランチをベースにしない。
2. **1 Task = 1 ブランチ = 1 PR = 1 Issue**。`Closes #XX` は最後の PR のみ、
   他は `Refs #XX`（同 Issue を複数 PR で Closes すると自動 close が壊れる）。
3. **push 前に必ず `git pull --rebase origin main`**。
4. **動作変更がある Task は CI を待ってからマージ**。

## サブエージェントの権限制限への備え

サブエージェントは `gh` / `npm` / `npx` の実行が許可なしで拒否されることがある
（CLAUDE.md の自律実行ルールが継承されない場合がある）。

**回避策**:
- **Haiku エージェントは「実装 + `git push` まで」で完了とする**。
  PR 作成・マージは親（Opus）がリカバリで実行する。
- 重要な判断を伴う `gh` / `npm install` は最初から Opus が直接やる方が早い。
- エージェントへの指示書末尾に必ず明記:
  「権限制限で進めない場合は実装と push だけ完了させて報告」
  「設計判断が必要になったら勝手に決めず止めて報告」

## Haiku に投げてよい / ダメ の早見表

| 投げてよい（機械的） | 投げてはダメ（判断あり） |
|---|---|
| 確定 I/F に対する関数実装 | I/F・型・データ構造の設計 |
| テスト追加（仕様が明確） | 外部 API 連携方針・CORS 回避設計 |
| curated データ・定数の追加 | ライブラリ選定・依存追加の判断 |
| CSS / マークアップの定型修正 | コンフリクト解消・アーキ変更 |
| ドキュメントの定型更新 | リリース判断・破壊的変更 |
